## Changes Made

### 1. SDK `getEntity` — required `type` param (`packages/admin-sdk/src/client.ts`)

Changed signature from `getEntity(id: string)` to `getEntity(id: string, params?: { type?: string })`.
When `params.type` is provided, it is forwarded as a `?type=<T>` query parameter, matching the backend contract exactly (`GET /admin/entities/<id>?type=<T>`).
When absent, no query param is sent (backwards-compatible for internal usage without type).

Updated `client.test.ts`:
- Renamed existing test to clarify it's the "no type" variant; added assertion that `query` is undefined.
- Added new test: `getEntity("abc", { type: "orders" })` → asserts `query.type === "orders"`.

### 2. Router generalization (`packages/admin-sdk/src/router.ts`)

The existing `matchRoute` implementation already supported multiple `:param` segments via its segment-by-segment loop — it was only the JSDoc comment that was wrong. Fixed the comment to document the real capability.

Added 3 new tests to `router.test.ts`:
- Two `:param` segments: `entities/:type/:id` matches `entities/orders/abc-123` → `{ type: "orders", id: "abc-123" }`.
- Negative: two-param route does not match one-segment or two-segment paths of wrong depth.
- Mixed static + two params: `admin/:type/:id` vs `entities/:type/:id` — correct first-match.

All 15 router tests (12 existing + 3 new) green.

### 3. Entity-browser: `type` as first-class (`plugins/entity-browser/`)

**`EntityList.tsx`:**
- `type` state is always present (existing behavior retained).

**`EntityDetail.tsx`:**

**`index.ts`:**

### 4. Contract fixture test (`apps/host/src/contract.test.ts`)

Exercises `FabriqClient` + `createHttpTransport` against a `fetchImpl` fake that returns the EXACT JSON shapes from `forgeext/adminapi`:

- `GET /admin/meta` → `{ "name": "fabriq-admin-api", "version": "0.1.0", "capabilities": ["entities.read"] }`
- `GET /admin/entities?type=orders` → `{ "items": [...], "nextCursor": "" }`
- `GET /admin/entities/<id>?type=orders` → `{"id","type","data"}`
- `GET /admin/entities` (no type) → 400
- `GET /admin/entities/unknown-id?type=orders` → 404

**Assertions (7 tests):**
1. `getMeta()` parses `name`/`version`/`capabilities` correctly.
2. `listEntities({type:"orders"})` returns `items[0].id`, `items[0].type`, `items[0].data`, reads `nextCursor: ""`.
3. The request URL actually included `type=orders` query param.
4. `listEntities({})` (no type) surfaces the 400 as a thrown error.
5. `getEntity(id,{type:"orders"})` returns the record.
6. The `getEntity` request URL included `type=orders`.
7. Unknown id → `getEntity` throws with 404.

All 7 contract tests green.
